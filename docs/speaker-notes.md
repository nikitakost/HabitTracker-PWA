# Нотатки доповідача: Habit Tracker PWA

Цей файл можна використовувати як готовий текст для захисту. Орієнтовна структура - 9 слайдів: достатньо, щоб пояснити суть проєкту, але не перетворити презентацію на лекцію.

---

## Слайд 1. Що це за проєкт

На слайді: назва `Habit Tracker PWA`, короткий опис, ключові слова `offline-first`, `sync`, `PWA`.

Текст виступу:

Наш проєкт - це full-stack застосунок для відстеження звичок, реалізований як Progressive Web App. Ідея проста: користувач створює звички, щодня відмічає прогрес і може продовжувати працювати навіть без інтернету.

Головний акцент тут не лише на списку звичок. Ми хотіли показати повноцінний вебзастосунок із нормальним UX, локальним збереженням, синхронізацією з сервером, авторизацією, тестами та зрозумілою архітектурою.

---

## Слайд 2. Навіщо тут PWA та offline-first

На слайді: проблема нестабільного інтернету, локальне збереження, подальша синхронізація.

Текст виступу:

Трекер звичок - це хороший приклад для offline-first підходу. Якщо користувач хоче відмітити звичку виконаною, застосунок не повинен ламатися лише через те, що зараз немає мережі.

Тому клієнт зберігає стан локально через Zustand persistence та LocalForage. Коли інтернет зникає, користувач все одно бачить свої звички і може продовжувати робити зміни. Коли з'єднання повертається, ці зміни синхронізуються з backend.

Для демонстрації ми додали явні статуси: `Offline mode`, `Saved locally`, `Sync pending`, `Syncing` і `Synced`. Завдяки цьому користувач не просто "вірить", що sync працює, а бачить, що саме відбувається.

---

## Слайд 3. Як працює PWA

На слайді: `manifest`, `service worker`, `app shell cache`, production build.

Текст виступу:

PWA-частина реалізована через `vite-plugin-pwa`. У застосунку є web manifest, який описує назву, іконки, тему та standalone-режим. Також генерується service worker, який кешує production assets.

Важлива деталь: offline-режим коректно тестується саме на production build, тобто після `npm run build` і запуску preview. Dev-сервер Vite підвантажує live-модулі, React Refresh і dev-client, тому він не є показником реальної offline-поведінки PWA.

Service worker відповідає за те, щоб оболонка застосунку відкривалася повторно без інтернету. А бізнес-дані, тобто звички та sync-state, зберігаються окремо в локальному storage.

---

## Слайд 4. Основний користувацький сценарій

На слайді: login, create habit, check-in, delete, sync.

Текст виступу:

Користувач спочатку входить у систему або створює акаунт. Після першого входу дані акаунта зберігаються локально, тому застосунок може відкриватися offline без зайвих запитів на `/auth/me`.

На головному екрані користувач створює звички, відмічає виконання за сьогодні, бачить прогрес за останні сім днів і отримує досягнення. На сторінці профілю винесена інформація, яка стосується саме акаунта: username, user id, дата створення та персональна статистика.

Якщо користувач видаляє звичку, ми більше не використовуємо простий браузерний `confirm`. Замість цього є власне модальне вікно. А саме видалення реалізоване як soft delete через `deletedAt`, щоб синхронізація була безпечнішою.

---

## Слайд 5. Синхронізація

На слайді: local state -> push -> server -> pull -> synced.

Текст виступу:

Синхронізація побудована навколо локального стану. Коли користувач створює, змінює або видаляє звичку, Zustand-store виставляє `hasPendingChanges=true`. UI одразу показує, що зміни збережені локально.

Якщо користувач online, auto-sync відправляє зміни на сервер. Також є кнопка `Sync now`, щоб під час демо можна було вручну показати момент синхронізації. Після успішного push і pull застосунок оновлює `lastSyncedAt` і показує статус `Synced`.

Якщо інтернет вимкнений, sync-запити не виконуються. Це важливо: застосунок не повинен нескінченно спамити `/sync` або `/auth/me`, коли мережі немає.

---

## Слайд 6. Soft delete

На слайді: `deletedAt`, tombstone, чому це краще за snapshot delete.

Текст виступу:

Раніше видалення можна було реалізувати через snapshot: якщо звички немає у payload, сервер її видаляє. Але для offline-first це ризиковано, бо різні пристрої можуть мати різний локальний стан.

Тому зараз використовується soft delete. Коли користувач видаляє звичку, вона не зникає з даних повністю, а отримує поле `deletedAt`. UI її приховує, але sync може передати серверу інформацію: "ця звичка була видалена".

Сервер зберігає tombstone і повертає його в pull-відповіді. Це допомагає іншим клієнтам також зрозуміти, що запис треба приховати, а не випадково відновити.

---

## Слайд 7. Архітектура frontend і backend

На слайді: frontend layers, backend flow `routes -> controllers -> services -> repositories -> prisma`.

Текст виступу:

Frontend організований у стилі Feature-Sliced Design: `app`, `pages`, `widgets`, `features`, `entities`, `shared`. Це допомагає розділити відповідальність. Наприклад, `entities/habit` містить доменний стан звичок, `features/sync-data` відповідає за синхронізацію, а `shared/ui` містить універсальні компоненти.

Backend побудований у Nest-style підході, але на Express. Запит проходить через route, потім controller, service, repository і Prisma. Controller працює з HTTP, service містить бізнес-логіку, repository відповідає за доступ до бази.

Такий поділ робить код простішим для тестування і пояснення, бо кожен шар має свою конкретну роль.

## Слайд 8. Стек і безпека

На слайді: React, TypeScript, Vite, Zustand, TanStack Query, Tailwind, Express, Prisma, SQLite, Zod, JWT.

Текст виступу:

На frontend використовується React з TypeScript і Vite. Для локального стану - Zustand, для серверних операцій - TanStack Query, для стилізації - Tailwind CSS.

На backend використовується Node.js, Express, TypeScript, Prisma та SQLite. Валідація запитів зроблена через Zod. Авторизація працює через JWT у cookie-based flow з `credentials: include`.

Також є CORS-конфігурація, rate limiting і централізована обробка помилок. Ліміти запитів винесені в environment variables, щоб їх можна було змінювати без редагування коду.

---

## Слайд 9. Тестування і перевірка

На слайді: unit tests, API tests, repository tests, frontend component/store tests, `npm run verify`.

Текст виступу:

Проєкт покритий тестами на кількох рівнях. На backend є тести контролерів, сервісів, репозиторіїв, middleware та інтеграційні HTTP-тести через Supertest. Це перевіряє не лише окремі функції, а й реальний API flow.

На frontend додані тести для базових UI-компонентів, модального підтвердження, header sync-state, auth offline fallback і Zustand-store. Окремо перевіряється, що видалення є soft delete, а локальні зміни позначаються як pending sync.

Для фінальної перевірки є одна команда: `npm run verify`. Вона запускає typecheck, build і тести по workspaces. Це зручно для здачі, бо можна швидко показати, що проєкт збирається і проходить перевірки.

## Короткий фінал

Якщо потрібно завершити виступ одним абзацом:

Цей проєкт демонструє не просто CRUD для звичок, а повноцінний offline-first PWA: з локальним збереженням, явним sync-status, ручною та автоматичною синхронізацією, soft delete, чистою frontend/backend архітектурою, авторизацією, документацією і тестами. Головна ідея - зробити застосунок, який залишається корисним навіть тоді, коли мережа нестабільна.
