# Release v1.0.0 - Habit Tracker PWA

Release date: 2026-04-20

## Summary

`v1.0.0` is the first stable release of the Habit Tracker PWA. The project is ready to present as a full-stack offline-first application with a React PWA client, a layered Node.js backend, Prisma persistence, authentication, synchronization, tests, CI, and documentation.

The main product idea is simple: users can create habits, track daily progress, keep working offline, and synchronize changes with the server when the connection returns.

## Product Highlights

- Offline-first habit tracking: habits stay available without network access.
- Installable PWA behavior: production assets are precached by the service worker.
- Local persistence: client state is stored through Zustand and LocalForage.
- Clear sync UX: the app shows whether changes are saved locally, pending, syncing, synced, or failed.
- Manual sync control: users can press `Sync now` during demos or real usage.
- Account profile: account identity and personal habit statistics live on a dedicated profile page.
- Achievements: progress milestones make the tracker feel more complete and demo-friendly.
- Safe deletion: habit deletion uses a custom dialog and soft delete tombstones instead of fragile snapshot deletion.

## Frontend

The frontend is built with React, TypeScript, Vite, Zustand, TanStack Query, Tailwind CSS, and `vite-plugin-pwa`.

The code is organized in a Feature-Sliced Design style:

- `app` initializes providers, routing, and global recovery behavior.
- `pages` compose full application screens.
- `widgets` contain larger UI sections such as the dashboard, header, achievements, and habit tracker.
- `features` own user scenarios such as authentication and synchronization.
- `entities` keep domain state for users and habits.
- `shared` contains UI primitives, API helpers, and utilities.

Important frontend behavior:

- The app avoids `/auth/me` checks when a saved user exists locally.
- Sync requests are not sent while offline.
- Habit mutations mark local state as pending sync.
- Soft-deleted habits are hidden immediately and cleaned from local storage only after successful sync.
- A React `ErrorBoundary` prevents a blank-screen failure and gives the user a reload recovery path.

## Backend

The backend is built with Node.js, Express, TypeScript, Prisma, SQLite, Zod, JWT, and Vitest.

The architecture follows a Nest-style responsibility split while staying on Express:

- `routes` define endpoints and middleware.
- `controllers` handle HTTP input and output.
- `services` contain business rules.
- `repositories` encapsulate Prisma access.
- `modules` wire concrete dependencies together.
- `infrastructure` exposes shared providers such as Prisma.
- `common` contains errors, middleware, and shared request types.

Sync data is separated into API DTOs and persistence mapping:

- `HabitSyncPayload` describes the push request body.
- `HabitDto` describes the pull response shape.
- mapper functions convert between Prisma records and API contracts.

## Offline And Sync Model

The release uses an eventual-consistency sync model:

1. User changes a habit locally.
2. Zustand persists the change and marks `hasPendingChanges=true`.
3. UI shows `Saved locally` or `Sync pending`.
4. When online, sync pushes local habits to `/api/sync/push`.
5. The app pulls latest server state from `/api/sync/pull`.
6. Successful sync updates `lastSyncedAt` and clears pending state.

Deletion is represented with `deletedAt`, so other clients can learn that a habit was deleted instead of accidentally restoring it from an older snapshot.

## Local Setup

Install dependencies:

```bash
npm install
```

Prepare Prisma and the local SQLite database:

```bash
npm run prisma:push --workspace=server
```

Run development servers:

```bash
npm run dev --workspace=client
npm run dev --workspace=server
```

Run production-style PWA preview:

```bash
npm run build --workspace=client
npm run preview --workspace=client
```

## Notes For Presentation

The strongest demo path for this release:

1. Open the app online and log in.
2. Create a habit and show the sync status.
3. Disable network.
4. Create or update a habit and show `Saved locally` / `Offline mode`.
5. Re-enable network.
6. Press `Sync now` or wait for auto-sync.
7. Show the status changing to `Synced`.

This demonstrates the main engineering value of the project: the app behaves like a real offline-first PWA instead of a basic online-only CRUD interface.
